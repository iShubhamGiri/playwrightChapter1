01 · Theory & Concepts

What is Playwright?

The modern standard for reliable, fast, and scalable browser automation.

Definition Playwright is an open-source Node.js library developed by Microsoft that provides a high-level API to control Chromium, Firefox, and WebKit browsers. It enables reliable end-to-end testing, web scraping, screenshot generation, and browser automation — all with a single consistent API.
🎭
What it does

Playwright automates real browser interactions — clicks, form fills, navigation, file downloads, API calls, and network interception — exactly as a real user would experience them.

🏢
Who made it

Created by Microsoft engineers (many of whom previously built Puppeteer at Google). Released in 2020, it has grown to be the dominant modern testing framework.

Why Playwright?

⚡
Auto-Wait

Playwright waits for elements to be actionable before interacting — no more sleep() hacks or flaky tests.

🌐
True Cross-Browser

One test runs on Chrome, Firefox, and Safari (WebKit). Real browsers, not headless-only simulations.

🚀
Built for Speed

Parallel test execution out of the box. Tests run in isolated browser contexts — faster than launching new browsers.

🔬
Network Control

Intercept, mock, and modify HTTP requests. Test API failures without touching your backend server.

🎯
Robust Locators

Semantic locators using getByRole, getByText, getByLabel — resilient to UI changes.

📸
Visual Testing

Built-in screenshot comparisons, video recording on failure, and Playwright Trace Viewer for debugging.

07 · Interview Prep

Interview Questions & Answers

Click any question to reveal the answer. Beginner to intermediate level.

BEGINNER What is Playwright and who created it? ▼
Playwright is an open-source browser automation framework created by Microsoft. It provides a unified API to control Chromium, Firefox, and WebKit browsers for end-to-end testing, web scraping, and automation. Released in 2020 by engineers who previously built Puppeteer at Google.
BEGINNER Why choose Playwright over Selenium? ▼
Playwright offers: (1) built-in auto-wait — no manual sleeps needed, (2) native parallel execution without Selenium Grid, (3) better performance via direct CDP/WebSocket connection, (4) built-in network interception, screenshot, and video recording, (5) simpler setup — one npm install vs complex WebDriver configuration, (6) modern TypeScript-first API with better IDE support.
BEGINNER What are locators in Playwright? ▼
Locators are Playwright's way of finding elements on a page. Unlike Selenium's findElement(), Playwright locators are lazy — only evaluated when you interact with them — and they auto-retry if the element isn't ready yet. Best practice locators: getByRole() (semantic), getByLabel() (forms), getByText() (visible text), getByTestId() (stable custom attribute).
BEGINNER What is auto-wait in Playwright? ▼
Auto-wait automatically waits for elements to be in an actionable state before performing interactions. Before a .click(), Playwright checks: attached to DOM → visible → stable (not animating) → enabled → not obscured. This eliminates flaky tests caused by race conditions. Default timeout: 30 seconds.
INTERMEDIATE How does parallel execution work in Playwright? ▼
Playwright runs tests in parallel at two levels: (1) Worker-level — multiple test files run simultaneously across CPU cores (configured via workers in playwright.config.ts). (2) Project-level — all configured browsers run in parallel. Each test gets its own BrowserContext — an isolated session — so tests don't interfere.
INTERMEDIATE What is a Browser Context? ▼
A BrowserContext is an isolated browser session — like an incognito window. Each context has its own cookies, localStorage, sessionStorage, HTTP auth, and network settings. Multiple contexts run in a single browser process simultaneously, making them much cheaper than launching separate browsers. Use case: test multi-user scenarios — Context A as Admin, Context B as Customer.
INTERMEDIATE What types of assertions does Playwright support? ▼
Two types: (1) Web-first assertions — e.g., expect(locator).toBeVisible() — automatically retry until condition is met or timeout. (2) Generic assertions — e.g., expect(value).toBe(true) — synchronous checks. Key web-first: toBeVisible(), toBeEnabled(), toHaveText(), toHaveURL(), toHaveTitle().
INTERMEDIATE What is the Page Object Model (POM)? ▼
POM is a design pattern where each web page/component has a class encapsulating its locators and actions. Benefits: (1) reduces code duplication, (2) improves maintainability — UI changes require updating only the Page class, (3) improves readability — test code reads like a user story: loginPage.login('admin', 'pass'). In Playwright, POM classes accept a Page object in the constructor.
